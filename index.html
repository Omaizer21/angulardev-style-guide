Architecture Overview

The core / shared / features / layout pattern for scalable standalone Angular apps.

🧠

core/

Singleton services, functional guards, functional interceptors, models, enums, constants, and utilities. Provided at the root — app-wide singletons.

  • Functional guards (auth, role, no-auth)
  • Functional interceptors (auth, error, loading)
  • Singleton services with inject()
  • Global models, enums, constants
  • Pure utility functions
🔗

shared/

Reusable standalone components, directives, and pipes used across multiple features. Import directly into feature components.

  • Reusable UI components (standalone)
  • Custom attribute directives
  • Custom pipes
  • Reactive form validators
🧩

features/

Lazy-loaded feature areas, each a distinct business domain. Each feature owns its pages, components, services, models, and routes — fully self-contained.

  • Auth (login, register, forgot-password)
  • Dashboard (stats, charts, activity)
  • Users (list, detail, edit)
  • Each feature exposes only its routes file
🖼️

layout/

Layout components forming the visual shell: header, sidebar, footer, and main layout wrapper that composes them with <router-outlet>.

  • Header with navigation & theme toggle
  • Collapsible sidebar navigation
  • Footer with links
  • Main layout composition component

Naming Conventions

Angular's consistent file naming conventions for every artifact type.

.component.ts

Components

UI building blocks. hero.component.ts → class HeroComponent → selector app-hero. Always standalone in Angular 17+.

.service.ts

Services

Business logic & data access. auth.service.ts → class AuthService. Use inject() over constructor injection.

.guard.ts

Guards

Route protection. auth.guard.ts → function authGuard. Angular 17+ uses functional guards via CanActivateFn.

.interceptor.ts

Interceptors

HTTP middleware. auth.interceptor.ts → function authInterceptor. Uses HttpInterceptorFn type.

.pipe.ts

Pipes

Template transforms. truncate.pipe.ts → class TruncatePipe. Mark as standalone: true.

.directive.ts

Directives

DOM behavior. highlight.directive.ts → class HighlightDirective. Mark as standalone: true.

.model.ts

Models

TypeScript interfaces/types. user.model.ts → interface User. Prefer interfaces over classes for data shapes.

.spec.ts

Tests

Unit tests co-located with source. hero.component.spec.ts tests HeroComponent. Use TestBed for Angular-specific testing.

.routes.ts

Routes

Route definitions. auth.routes.ts → export AUTH_ROUTES. Use loadComponent and loadChildren.

.resolver.ts

Resolvers

Pre-load route data. user.resolver.ts → function userResolver. Uses ResolveFn<T> functional type in Angular 17+.

.enum.ts

Enums

Enumerated types. role.enum.ts → enum Role. Consider as const objects for tree-shakable alternatives.

.constant.ts

Constants

App-wide constants. api-endpoints.constant.ts → export API_ENDPOINTS. Use as const for type narrowing.

.util.ts

Utilities

Pure helper functions. date.util.ts → named exports. No Angular dependencies — easily unit-testable.

Key Principles

The design principles behind a well-architected Angular application.

S

Single Responsibility

Each file and class has one reason to change. Components handle UI; services handle logic; resolvers handle data loading.

O

Open / Closed

Design services and components to be extended without modifying existing code. Use interfaces and DI tokens.

L

Liskov Substitution

Derived classes should be substitutable for their base types. Use interfaces to define contracts between layers.

I

Interface Segregation

Prefer many small, specific interfaces. Keep service contracts lean — avoid god-service interfaces.

D

Dependency Inversion

Depend on abstractions via inject() and DI tokens. Angular's DI system naturally enforces this.

🔄 DRY — Don't Repeat Yourself

Extract shared logic into services, pipes, directives, and utility functions. The shared/ folder is your best friend for UI; core/utils for business logic.

📦 Lazy Loading

Use loadComponent() and loadChildren() in routes to keep the initial bundle small. Every feature should be lazy by default.

🗂️ Feature Encapsulation

Features own their pages, components, services, and models. Only the routes file is exposed to the rest of the app — no leaking internals.

🧪 Co-located Tests

Spec files live next to the code they test (*.spec.ts). This makes finding and maintaining tests natural — no separate test directory.

State Management

Choose the right tool: Signals for local/simple state, NgRx for complex shared state.

⚡
Signals Recommended
Angular 16+ · Built-in · Simple & Medium apps

Reactive primitives built into Angular. Zero dependencies, fine-grained reactivity, and seamless integration with OnPush change detection. Best for local UI state and simple shared services.

// counter.service.ts
import { Injectable, signal, computed } from '@angular/core';

@Injectable({ providedIn: 'root' })
export class CounterService {
  private count = signal(0);
  doubleCount = computed(() => this.count() * 2);

  increment() { this.count.update(v => v + 1); }
  decrement() { this.count.update(v => v - 1); }
  reset() { this.count.set(0); }
}

// component.ts
@Component({
  standalone: true,
  template: `
    <p>Count: {{ counter.count() }}</p>
    <p>Double: {{ counter.doubleCount() }}</p>
    <button (click)="counter.increment()">+</button>
  `
})
export class CounterComponent {
  counter = inject(CounterService);
}
🏪
NgRx Store Large Apps
@ngrx/store · Redux pattern · Devtools support

Predictable, immutable state container with time-travel debugging. Ideal for complex, cross-feature state shared by many components with side effects.

// counter.actions.ts
export const increment = createAction('[Counter] Increment');
export const decrement = createAction('[Counter] Decrement');

// counter.reducer.ts
export const counterReducer = createReducer(
  { count: 0 },
  on(increment, state => ({ ...state, count: state.count + 1 })),
  on(decrement, state => ({ ...state, count: state.count - 1 }))
);

// component.ts
@Component({ standalone: true, imports: [AsyncPipe] })
export class CounterComponent {
  private store = inject(Store);
  count$ = this.store.select(selectCount);

  increment() { this.store.dispatch(increment()); }
}
📋
Service + Observables Classic
BehaviorSubject · RxJS · Medium apps

A proven pattern using BehaviorSubject for state and exposing observables. Still valid for teams familiar with RxJS, though Signals are the modern replacement.

// auth.service.ts (classic BehaviorSubject pattern)
@Injectable({ providedIn: 'root' })
export class AuthService {
  private http = inject(HttpClient);
  private userSubject = new BehaviorSubject<User | null>(null);

  user$ = this.userSubject.asObservable();
  isLoggedIn$ = this.user$.pipe(map(u => u !== null));

  login(email: string, pass: string): Observable<User> {
    return this.http.post<User>('/api/login', { email, pass }).pipe(
      tap(user => this.userSubject.next(user))
    );
  }

  logout() { this.userSubject.next(null); }
}
🔧
When to Choose What
Decision guide
  • 🟢 Signals — Local UI state, small apps, component-level reactivity
  • 🔵 Service+Signals — Shared state across a few components, medium apps
  • 🟠 NgRx — Complex state, many features, need DevTools & time-travel debug
  • 🔴 NgRx+Nx — Multiple apps sharing state, enterprise monorepo, team isolation

Testing Strategy

A three-layer testing pyramid: unit tests for logic, integration for wiring, E2E for user journeys.

🧪
Unit Testing Jasmine / Jest

Test services, pipes, and utilities in isolation. Use TestBed for Angular-specific DI. Aim for 70%+ coverage on business logic.

describe('AuthService', () => {
  let service: AuthService;

  beforeEach(() => {
    TestBed.configureTestingModule({
      providers: [
        AuthService,
        provideHttpClientTesting()
      ]
    });
    service = TestBed.inject(AuthService);
  });

  it('should return user on login', fakeAsync(() => {
    const mockUser = { id: 1, name: 'Test' };
    service.login('u', 'p').subscribe(u => {
      expect(u).toEqual(mockUser);
    });
    TestBed.inject(HttpTestingController)
      .expectOne('/api/login').flush(mockUser);
    tick();
  }));
});
🔧
Component Testing Component Harness

Test components with Angular's TestBed and component harnesses. Verify rendering, inputs, outputs, and user interactions.

describe('ButtonComponent', () => {
  let fixture: ComponentFixture<ButtonComponent>;
  let loader: HarnessLoader;

  beforeEach(async () => {
    await TestBed.configureTestingModule({
      imports: [ButtonComponent]
    }).compileComponents();
    fixture = TestBed.createComponent(ButtonComponent);
    loader = TestbedHarnessEnvironment.loader(fixture);
  });

  it('should emit click', async () => {
    let clicked = false;
    fixture.componentInstance.clicked.subscribe(() => clicked = true);
    const btn = await loader.getHarness(MatButtonHarness);
    await btn.click();
    expect(clicked).toBe(true);
  });
});
🎬
E2E Testing Playwright

Simulate real user journeys in a browser. Use data-testid attributes for stable selectors that survive refactoring.

// login.spec.ts (Playwright)
test('user can login and see dashboard', async ({ page }) => {
  await page.goto('/login');

  await page.fill('[data-testid=email]', 'user@example.com');
  await page.fill('[data-testid=password]', 'password123');
  await page.click('[data-testid=submit]');

  await expect(page).toHaveURL('/dashboard');
  await expect(page.locator('h1'))
    .toContainText('Dashboard');
});
📊
Testing Pyramid

The optimal mix for Angular applications:

  • 🟢 70% Unit — Services, pipes, utilities, reducers
  • 🔵 20% Integration — Components with TestBed, routing
  • 🔴 10% E2E — Critical user journeys only
Tip: Always use data-testid attributes for E2E selectors. Avoid selecting by CSS classes or text content.

Performance Tips

Practical techniques for fast, efficient Angular 17+ applications.

⚡

OnPush + Signals

Use ChangeDetectionStrategy.OnPush on all components. With signals, Angular automatically tracks dependencies — only rerenders what changed.

@Component({
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    @for (item of items(); track item.id) {
      <app-item [data]="item" />
    }
  `
})
export class ListComponent {
  private svc = inject(DataService);
  items = this.svc.items; // Signal<Item[]>
}
📦

Lazy Loading

Every feature should be lazy-loaded. Use loadComponent() for single components or loadChildren() for feature route groups.

// app.routes.ts
export const routes: Routes = [
  {
    path: 'dashboard',
    loadComponent: () =>
      import('./features/dashboard/dashboard.component')
        .then(m => m.DashboardComponent)
  },
  {
    path: 'users',
    loadChildren: () =>
      import('./features/users/users.routes')
        .then(m => m.USERS_ROUTES)
  }
];
📊

track in @for

The new @for control flow requires a track expression. Angular uses it to minimize DOM operations. Always track by a unique identifier.

@for (user of users(); track user.id) { <app-user-card [user]="user" /> } @for (tag of tags(); track tag) { <span class="tag">{{ tag }}</span> } @for (item of items(); track item.id; let i = ) { <li>{{ i + 1 }}. {{ item.name }}</li> }
💰

Bundle Optimization

Reduce initial bundle size with these build-time strategies.

  • ✅ Use ng build --configuration production
  • ✅ Enable esbuild builder (default in Angular 17)
  • ✅ Analyze bundle: ng build --stats-json
  • ✅ Avoid barrel imports from large libraries
  • ✅ Use image optimization with NgOptimizedImage
🔄

Defer Blocks

Angular 17's @defer enables fine-grained lazy loading of template sections, improving LCP and reducing TTI.

@defer (on viewport) { <app-heavy-chart [data]="chartData" /> } @placeholder { <div class="chart-skeleton"></div> } @loading (minimum 300ms) { <app-spinner /> }
🖼️

Image Optimization

Use Angular's built-in NgOptimizedImage directive for automatic lazy loading, responsive sizes, and Core Web Vitals improvements.

@Component({
  standalone: true,
  imports: [NgOptimizedImage],
  template: `
    <img ngSrc="hero.jpg" width="800"
         height="400" priority />

    <img ngSrc="photo.jpg" width="400"
         height="300" loading="lazy" />
  `
})
export class HeroComponent {}

Common Anti-patterns

Mistakes to avoid — and the Angular 17+ idiomatic alternatives.

1. Manual Subscription in Template

❌ Don't
// Subscribing in component
// then binding local variable
ngOnInit() {
  this.data$.subscribe(
    d => this.data = d
  );
}
✅ Do
// Use async pipe or signals
data$ = this.svc.data$;
// template:
// *ngIf="data$ | async as data"
// Or: data = toSignal(data$);

2. Logic in Constructor

❌ Don't
constructor(
  private svc: UserService
) {
  // Side effects in ctor!
  this.svc.loadUsers();
  this.initForm();
}
✅ Do
private svc = inject(UserService);

ngOnInit() {
  // Side effects in ngOnInit
  this.svc.loadUsers();
  this.initForm();
}

3. Memory Leaks — Unmanaged Subscriptions

❌ Don't
ngOnInit() {
  // Never unsubscribed!
  this.svc.data$.subscribe(
    data => this.data = data
  );
}
✅ Do
data$ = this.svc.data$
  .pipe(takeUntilDestroyed());
// Or use async pipe
// Or convert to signal:
data = toSignal(this.svc.data$);

4. Overusing any

❌ Don't
// Loses all type safety
processData(data: any) {
  this.user = data.user;
  this.count = data.count;
}
✅ Do
interface ApiData {
  user: User;
  count: number;
}
processData(data: ApiData) {
  this.user = data.user;
  this.count = data.count;
}

5. Direct DOM Manipulation

❌ Don't
// Bypasses Angular & SSR
document.getElementById('btn')
  ?.classList.add('active');
element.nativeElement
  .style.color = 'red';
✅ Do
// Use Angular bindings
isActive = signal(false);
// template:
// [class.active]="isActive()"
// Or Renderer2 for dynamic:
this.renderer.addClass(
  el, 'active');

6. Fat Components

❌ Don't
// Component does everything
export class UserListComponent {
  users: User[] = [];
  // HTTP calls in component
  // Sorting, filtering logic
  // Form validation logic
  // ...
✅ Do
// Thin component + service
export class UserListComponent {
  private svc = inject(UserService);
  users = this.svc.users; // signal
  // Component only handles UI
  // All logic in service
}

7. Nested Subscriptions (Callback Hell)

❌ Don't
this.auth$.subscribe(user => {
  this.userSvc.get(user.id)
    .subscribe(profile => {
      this.posts$.subscribe(...)
    });
});
✅ Do
this.auth$.pipe(
  switchMap(user =>
    this.userSvc.get(user.id)),
  switchMap(profile =>
    this.getPosts(profile.id))
).subscribe(posts => ...);

8. Default Change Detection

❌ Don't
// No strategy specified
// Angular checks ALL components
// on every event & async op
@Component({
  standalone: true,
  template: `...`
})
export class ItemComponent {}
✅ Do
@Component({
  standalone: true,
  changeDetection:
    ChangeDetectionStrategy.OnPush,
  template: `...`
})
export class ItemComponent {
  // Use signals or immutable data
}

Migration Guide

Step-by-step migration paths to Angular 17+ modern patterns.

1
NgModules → Standalone Components

Run the built-in migration schematic: ng generate @angular/core:standalone. Then convert AppModule to use bootstrapApplication.

BEFORE (NgModule)
// app.module.ts
@NgModule({
  declarations: [
    AppComponent,
    HomeComponent
  ],
  imports: [
    BrowserModule,
    RouterModule.forRoot(routes)
  ],
  bootstrap: [AppComponent]
})
export class AppModule {}

// main.ts
platformBrowserDynamic()
  .bootstrapModule(AppModule);
AFTER (Standalone)
// main.ts
bootstrapApplication(AppComponent, {
  providers: [
    provideRouter(routes),
    provideHttpClient(
      withInterceptors([authInterceptor])
    )
  ]
});

// app.component.ts
@Component({
  standalone: true,
  imports: [RouterOutlet],
  template: `<router-outlet />`
})
export class AppComponent {}
2
*ngIf / *ngFor → @if / @for

Run the built-in migration: ng generate @angular/core:control-flow. New syntax is more readable and doesn't need CommonModule.

BEFORE (*ngIf / *ngFor)
<div *ngIf="user; else loading">
  {{ user.name }}
</div>
<ng-template #loading>
  Loading...
</ng-template>

<div *ngFor="let item of items;
     trackBy: trackById">
  {{ item.name }}
</div>

<div [ngSwitch]="status">
  <span *ngSwitchCase="'active'">
    Active</span>
  <span *ngSwitchDefault>
    Inactive</span>
</div>
AFTER (@if / @for / @switch)
@if (user) {
  <div>{{ user.name }}</div>
} @else {
  <div>Loading...</div>
}

@for (item of items; track item.id) {
  <div>{{ item.name }}</div>
} @empty {
  <p>No items found.</p>
}

@switch (status) {
  @case ('active') { <span>Active</span> }
  @default { <span>Inactive</span> }
}
3
BehaviorSubject → Signals

Replace BehaviorSubject state in services with signal(). Use toSignal() as a bridge for existing observables.

BEFORE (BehaviorSubject)
@Injectable({ providedIn: 'root' })
export class AuthService {
  private userSubject =
    new BehaviorSubject<User | null>(null);

  user$ = this.userSubject.asObservable();

  isLoggedIn$ = this.user$.pipe(
    map(u => u !== null)
  );

  setUser(user: User) {
    this.userSubject.next(user);
  }

  logout() {
    this.userSubject.next(null);
  }
}
AFTER (Signals)
@Injectable({ providedIn: 'root' })
export class AuthService {
  private userSignal =
    signal<User | null>(null);

  user = this.userSignal.asReadonly();

  isLoggedIn = computed(
    () => this.userSignal() !== null
  );

  setUser(user: User) {
    this.userSignal.set(user);
  }

  logout() {
    this.userSignal.set(null);
  }
}
4
Constructor Injection → inject()

The inject() function is the modern approach — works in constructors, factory functions, guards, and interceptors.

BEFORE (Constructor DI)
@Component({ standalone: true })
export class UserListComponent
  implements OnInit {

  users: User[] = [];

  constructor(
    private userService: UserService,
    private router: Router,
    private store: Store
  ) {}

  ngOnInit() {
    this.userService.getUsers()
      .subscribe(u => this.users = u);
  }
}
AFTER (inject() function)
@Component({ standalone: true })
export class UserListComponent
  implements OnInit {

  private userService = inject(UserService);
  private router = inject(Router);
  private store = inject(Store);

  users = signal<User[]>([]);

  ngOnInit() {
    this.userService.getUsers()
      .pipe(takeUntilDestroyed())
      .subscribe(u => this.users.set(u));
  }
}
